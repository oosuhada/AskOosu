# Product Build Loop

## 목적

아이디어를 포트폴리오 프로젝트나 서비스형 데모로 바꾸는 반복 루프입니다. 문제 정의, 사용자 시나리오, 범위 설정, 프로토타입, 통합, 리뷰, 문서화, 반복 개선으로 구성됩니다.

## 운영 규칙

- 기능보다 사용자가 겪는 문제를 먼저 정의한다.
- AI 기능은 실제 UX 흐름과 연결될 때만 넣는다.
- 가짜 진행률, 검증되지 않은 수치, 과장된 savings/source count를 피한다.
- 로딩, 실패, 빈 결과, 성공 상태를 함께 설계한다.
- 완성 후 project fact, story, decision log, postmortem, FAQ, guardrail을 문서화한다.

## 답변 활용 방향

우수의 작업 방식은 “빠르게 많이 만드는 것”보다 “문제 정의에서 검증과 문서화까지 연결하는 것”으로 설명합니다.

## 피해야 할 표현

- 기능이 많을수록 좋은 포트폴리오입니다.
- AI 기능은 있으면 무조건 좋습니다.
- 완성도보다 속도가 중요합니다.

## 연결 질문

- 우수는 어떻게 제품을 만드나요?
- 아이디어를 실제 프로젝트로 바꾸는 방식은?
- 포트폴리오 프로젝트에서 가장 중요하게 보는 기준은?

## 검토 메모

도구 선호도, 프로젝트 구조, 공개 포지셔닝이 바뀌면 이 문서를 함께 갱신해야 합니다.
